# Система нормирования трудоёмкости сборки ШАУ

Прогнозирование трудоёмкости сборки и базовой пусконаладки электрических шкафов
управления (ШАУ) в нормо-часах по спецификации (BOM) и параметрам проекта.

Решение построено как **три слоя**, которые работают независимо и дополняют друг друга:

| Слой | Компонент | Файл | Зависит от данных |
|------|-----------|------|-------------------|
| **A. Параметрическая модель** | Калькулятор норм по эвристикам | `Калькулятор_норм_ШАУ.xlsx` | нет (работает сразу) |
| **B. Извлечение признаков** | Парсер спецификаций | `parse_spec.py` | нет |
| **C. ML-модель** | Градиентный бустинг (план) | — | да (~150+ проектов) |

Слой A даёт рабочий инструмент с первого дня и служит baseline. Слой B автоматизирует
сбор датасета. Слой C обучается, когда накоплены фактические часы из 1С, и уточняет
оценку слоя A.

## Слой B — парсер спецификаций (`parse_spec.py`)

Извлекает из проектной документации (PDF-экспорт EPLAN / аналогичной CAD-системы)
плоский вектор признаков согласно разделу 2 ТЗ.

Поддерживаемые форматы (определяются по расширению): **PDF**, **Excel** (.xlsx/.xls —
выгрузка EPLAN «Перечень изделий / Parts list / Stückliste»), **CSV**.

### Запуск

```bash
python3 parse_spec.py "spec.pdf"                      # PDF
python3 parse_spec.py "bom.xlsx" --bom                # Excel + показать строки BOM
python3 parse_spec.py "bom.xlsx" --sheet "Лист1"      # указать лист Excel
python3 parse_spec.py "bom.csv"  --json features.json # CSV + сохранить вектор
```

**PDF vs табличный BOM**: из PDF извлекаются все признаки (включая число вводов, аналог.
цепи, интерфейсы — со схем). Из Excel/CSV — только то, что есть в перечне изделий
(аппараты, корпус, IP); число отверстий ввода и аналоговые цепи со схемы недоступны и
помечаются как нулевые. Проверено: по полям из BOM PDF и EPLAN-Excel дают идентичные
счётчики.

### Что и откуда извлекается

| Источник в документе | Признаки |
|----------------------|----------|
| Спецификация (BOM) | количество аппаратов по категориям, тип/габариты корпуса, IP |
| Схема расположения | число отверстий ввода (Ø и количество) |
| Таблица клемм | аналоговые цепи, интерфейсы |
| Схема электрическая | интерфейсы (Ethernet/RS-485/Modbus) |

### Схема вектора признаков

| Поле | Тип | Описание |
|------|-----|----------|
| `ip_rating` | str | степень защиты (IP21/54/65/66) |
| `width_mm`, `height_mm`, `depth_mm` | int | габариты корпуса |
| `mount_panel` | bool | наличие монтажной панели |
| `din_inclined` | bool | наклонная DIN-рейка |
| `cable_entries` | int | число кабельных вводов/отверстий |
| `cable_entries_detail` | list | детализация `[{count, dia_mm}]` |
| `breakers_le63` | int | автоматы до 63 А |
| `load_switches` | int | выключатели нагрузки / рубильники |
| `contactors` | int | контакторы / пускатели |
| `rcd` | int | УЗО / дифавтоматы |
| `freq_converters` | int | преобразователи частоты (ПЧ) |
| `soft_starters` | int | устройства плавного пуска (УПП) |
| `plc` | int | ПЛК / программируемые реле |
| `hmi` | int | HMI-панели |
| `power_supplies` | int | блоки питания |
| `signal_lamps` | int | сигнальные лампы / индикаторы |
| `relays` | int | промежуточные реле |
| `phase_relays` | int | реле контроля фаз |
| `analog_4_20mA`, `analog_0_10V` | int | аналоговые цепи (оценка по схеме) |
| `interfaces` | dict | `{ethernet, rs485, modbus, profibus}` |
| `terminals` | int | клеммы (Klemsan/Phoenix) |
| `fuse_holders` | int | держатели предохранителей |
| `fans`, `thermostats` | int | вентиляция, термостаты |

### Как устроено распознавание

1. **BOM**: принимаются только строки спецификации, начинающиеся с номера позиции и
   заканчивающиеся структурой `<кол-во> <заказной №> <изготовитель>`. Это отсекает
   текст со схемных листов.
2. **Категоризация**: по словарю `KEYWORDS` (ключевые слова → категория). Аксессуары
   (цоколь, скоба, изолятор) исключаются из счёта аппаратов списком `ACCESSORY`.
3. **Корпус / отверстия / аналоговые / интерфейсы**: регулярные выражения по полному
   тексту.

### Адаптация под свою номенклатуру

Парсер настраивается без изменения логики — правьте два словаря в начале файла:

```python
KEYWORDS = { "contactor": ["контактор", "пускатель", "кми", "lc1", ...], ... }
ACCESSORY = ["цоколь", "скоба", "изолятор", ...]
```

Добавьте обозначения и модели, принятые на предприятии. Для форматов, отличных от PDF
(Excel, XML EPLAN), замените только функцию загрузки текста `build_features()` —
остальная логика признаков переиспользуется.

### Известные ограничения

- `analog_4_20mA` оценивается по упоминаниям «4-20 мА» на схеме и может завышаться;
  точное число контуров берётся из таблицы клемм.
- `fans` считает элементы BOM (вентилятор + решётка = 2), а не вентиляционные узлы.
- Качество распознавания BOM зависит от того, насколько структурирован PDF-экспорт.

## Веб-интерфейс (`app.py`)

Локальный веб-инструмент с четырьмя разделами. Зависимостей сверх стандартной
библиотеки нет (Flask не требуется); графики рисуются на чистом SVG и работают офлайн.

```bash
python3 app.py                 # затем открыть http://127.0.0.1:8000
python3 app.py --port 8080
```

Разделы:

- **Прогноз** — загрузка спецификации (PDF / EPLAN-Excel / CSV) → трудоёмкость с
  разбивкой по этапам. Признаки можно поправить вручную и пересчитать.
- **Обучение** — кнопка запускает обучение ML-модели (CatBoost или numpy-fallback) прямо
  из интерфейса: метрики MAE/MAPE/R², сравнение с baseline, Anchor-тест, важности
  признаков, график «факт vs прогноз». После обучения прогноз сразу получает ML-поправку.
- **Аналитика** — распределение трудоёмкости, диаграмма «факт vs параметрическая оценка»,
  средние часы по IP и типу ПЛК, драйверы трудоёмкости (корреляции).
- **Статистика** — сводные показатели (число проектов, медиана/среднее/разброс часов,
  MAPE baseline) и разрезы по категориям.

Движок прогноза: параметрическая модель (всегда); после обучения или при наличии
`model_shau.cbm` (CatBoost) — добавляется ML-поправка к baseline.

## Обучение на реальных данных из 1С

Замкнутый цикл «спецификации + факт-часы → обученная модель»:

1. Заполните **`Шаблон_выгрузки_1С.xlsx`** (лист «Выгрузка 1С»): по строке на проект —
   имя файла спецификации, фактические нормо-часы из наряда-заказа, признак выброса.
   Подробности — на листе «Инструкция».
2. Сложите файлы спецификаций в папку `specs/`.
3. Соберите датасет и обучите:

```bash
python3 prepare_training_data.py --hours Шаблон_выгрузки_1С.xlsx --specs-dir specs/ --out training_data.csv
python3 train_model.py --data training_data.csv
```

`prepare_training_data.py` парсит признаки из каждой спецификации, присоединяет факт-часы
по ключу «Файл спецификации», отсеивает помеченные выбросы и выдаёт CSV в формате
`synthetic_dataset.csv` — готовый для `train_model.py`.

## Слой A — параметрический калькулятор (`Калькулятор_норм_ШАУ.xlsx`)

Прозрачная модель: трудоёмкость = сумма нормативов по этапам, умноженная на коэффициенты.

### Формула

```
T = Мехобработка + Силовой_монтаж + Слаботочка + Клеммы_маркировка + ПНР
    + Надбавка_IP66
```

где каждый этап = Σ(Кол-во признака × Норматив, ч/ед).

### Структура листа

- **Жёлтые ячейки (синий шрифт)** — вводимые: количество признаков и нормативы.
- **Чёрные ячейки** — формулы (`=Кол-во × Норматив`), подытоги, итог.
- Этапы: мехобработка → силовой монтаж → слаботочка → клеммы/маркировка → ПНР.
- Блок коэффициентов: надбавка IP66 (+30 % к мехобработке), нелинейный каскад ПЧ.

### Учёт нелинейностей (§8 ТЗ)

- **Каскад ПЧ**: при ≥2 ПЧ ПНР = 3 ч (настройка шины Modbus + ПЛК), а не `n × 0,5`.
- **IP66**: +30 % к мехобработке (сальники, уплотнители).
- **Аналоговые цепи**: отдельный норматив на экранирование/разделку/заземление и
  на калибровку при ПНР.

### Калибровка

Нормативы подобраны так, чтобы **эталонный (Anchor) шкаф = 20 ч** (требование ТЗ:
18–22 ч). По приложенному примеру калькулятор даёт **20,3 ч**.

> При внедрении нормативы уточняются по фактическим часам из 1С: достаточно изменить
> значения в жёлтых ячейках столбца «Норматив, ч/ед».

## Слой C — ML-модель (реализованный прототип)

Прототип конвейера обучения реализован и проверяется на синтетических данных.

### Запуск

```bash
python3 make_dataset.py --n 800 --out synthetic_dataset.csv   # сгенерировать датасет
python3 train_model.py  --data synthetic_dataset.csv          # обучить и оценить
```

### Что делает `train_model.py`

1. **Движок**: CatBoost (основной). Если он не установлен — автоматически numpy-fallback
   (`gbdt_numpy.py`), чтобы конвейер запускался в любом окружении.
2. **Очистка выбросов** (ТЗ, Этап 1): отсев записей, где факт расходится с оценкой не
   из-за сложности (простои поставок и т.п.).
3. **Обучение на остатках**: цель = `факт − baseline`, модель учит поправку к оценке
   слоя A — устойчивее на малых данных.
4. **Weighted loss**: повышенный вес редким сложным шкафам (IP66, каскад ПЧ, модульный ПЛК).
5. **Оценка**: MAE / MAPE / R², сравнение с baseline, тест на Anchor, важности признаков.

### Результат на синтетике (numpy-fallback; CatBoost ≥ не хуже)

| Модель | MAE, ч | MAPE | R² |
|--------|-------:|-----:|---:|
| Параметрическая (Слой A) | 0,88 | 4,9 % | 0,946 |
| **Слой A + ML-поправка** | **0,70** | **4,1 %** | **0,966** |

Тест на Anchor: 21,5 ч (диапазон ТЗ 18–22 ч). Критерии ТЗ (MAPE ≤ 15 %, R² ≥ 0,85) —
соблюдены. ML улучшает baseline за счёт структурных нелинейностей (IP66×вводы, каскад
ПЧ + модульный ПЛК, нелинейный рост наладки аналоговых цепей).

> Данные синтетические — для отладки конвейера. На проде `synthetic_dataset.csv`
> заменяется выгрузкой факт-часов из 1С; признаки собираются парсером из архива проектов.

### Конвейер данных

```
Архив проектов (PDF / EPLAN-Excel)
        │  parse_spec.py
        ▼
   Вектор признаков ──┐
                      ├──►  Датасет  ──►  CatBoost  ──►  Прогноз, ч
Факт-часы из 1С ──────┘   (очистка     (на остатках
(наряд-заказы)            выбросов)     baseline)
```
